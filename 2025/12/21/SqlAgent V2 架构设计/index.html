SqlAgent V2 架构设计文档 (Hybrid RAG)

Created2025-12-21|Updated2025-12-21|Project Documentation
|Post Views:

1. 设计目标 (Design Goal)

在 V1 版本跑通线性链路的基础上，V2 版本 (Hybrid RAG) 的核心目标是解决 Context Window 瓶颈 和 语义理解不足 的问题，使系统能够支持企业级（500+张表）的数据库规模。

  • 核心功能：引入 混合检索 (Hybrid Search) 机制，结合 语义检索 (Vector) 和 关键词检索 (BM25)，实现 Schema 的按需加载（Schema Linking）。
  • 架构验证：验证 Qdrant 向量数据库在结构化元数据（DDL）检索场景下的有效性，以及 Embedding 模型对数据库字段的语义理解能力。
  • 非功能目标：将 Schema 检索的准确率（Recall@5）提升至 90% 以上，同时保持单轮对话延迟在可接受范围内（< 3s）。

2. 系统上下文 (System Context)

V2 版本在原有架构基础上，新增了 知识入库 (Ingestion) 离线流程和 向量知识库 (Knowledge Base) 组件。系统分为“离线准备”和“在线服务”两大部分。

3. 核心状态定义 (Agent State Definition)

V2 的 State 在 V1 基础上进行了扩展，主要增加了检索过程的中间元数据，用于 Debug 和 RAG 效果分析。

字段名 (Field) 类型 (Type) 描述 (Description) 更新来源节点 变更说明
user_query str 用户的原始问题 Input 不变
retrieved_schemas str 筛选后的 Top-N DDL 文本 Retrieve Node 逻辑变更：不再是全量，而是RAG结果
retrieved_table_names List[str] 召回的表名列表（如 ["t_user", "t_order"]） Retrieve Node 新增：用于调试检索准确率
generated_sql str LLM 生成的 SQL Generate Node 不变
sql_result List[Dict] SQL 执行结果 Execute Node 不变

4. 图结构拓扑 (Graph Topology)

V2 版本维持了 DAG (有向无环图) 结构。核心变化在于 Retrieve Node 的内部逻辑从“全量反射”变更为“混合检索”。

节点详细说明 (V2 Update)：

  1. Retrieve Node (Hybrid Search):
  • 输入: user_query
  • 逻辑:
  1. Vector Search: 将 query 转向量，在 Qdrant 中搜索语义相关的表（如搜“营收”匹配 finance_table）。
  2. Keyword Search (BM25): 对 query 分词，匹配表名/字段名（如搜“user”匹配 t_user）。
  3. RRF Fusion: 对两路结果进行加权融合，取 Top-5。
  4. Format: 从 MySQL 或缓存中获取这 5 张表的完整 DDL。
  • 输出: 更新 retrieved_schemas 和 retrieved_table_names。
  1. Generate Node: (逻辑不变，但输入的 Context 变短了，Token 利用率更高)
  2. Execute Node: (逻辑不变)

5. 运行时序 (Runtime Sequence)

V2 引入了 Embedding 和 Vector DB 的交互，时序变得更加丰富。

6. 关键技术决策 (Key Decisions)

6.1 向量数据库选型 (Qdrant)

  • 决策：使用 Qdrant。
  • 理由：
  1. 轻量级：支持 Docker 快速部署，也有 Python 内存模式（适合开发）。
  2. 混合检索支持：原生支持 Payload 过滤和稀疏向量（Sparse Vectors），方便后续做 BM25 优化。
  3. LangChain 集成：langchain-qdrant 库支持完善。

6.2 索引粒度 (Indexing Strategy)

  • 决策：以 “Table (表)” 为单位进行索引。
  • 理由：
  • 虽然用户查的是“字段”，但 SQL 生成的基础单元是“表”。
  • Document 格式设计：
  • 这样可以保证检索结果直接对应 DDL，无需二次聚合。

6.3 Embedding 模型

  • 决策：使用阿里云 DashScope (text-embedding-v2 或类似) 或 HuggingFace 本地模型 (bge-m3)。
  • 理由：支持中文语义，且维度适中（1024/1536），性价比较高。

7. 接口定义 (API Interface)

保持 V1 接口契约不变，实现向下兼容。

POST /api/v1/chat

  • Request: model 参数后续可用于指定 RAG 的检索策略（如 hybrid, vector_only）。
  • Response: data 结构保持不变。

8. 后续规划 (Roadmap)

V2 完成后，Schema 检索问题将得到解决，接下来的重点是准确率和复杂场景：

  • V3 Self-Correction: 引入 SQL 执行报错后的自动修正循环（Loop）。
  • V3 SQL Validator: 引入 sqlglot 进行 AST 级别的语法检查。
  • V4 Tools & Memory: 引入多轮对话记忆和联网搜索工具。
