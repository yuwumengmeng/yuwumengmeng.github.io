SqlAgent 项目介绍

Created2026-04-17|Updated2026-04-17|项目文档
|Post Views:

1. 项目概述

SqlAgent 是一个面向自然语言数据分析场景的 AI Agent 系统，目标是把“提问 → 检索 → 生成 SQL → 执行 → 修复 → 分析/可视化”串成可观测、可扩展、可复用的闭环。

项目已经整体完成，系统能力覆盖多智能体协作、分层 Schema + Value Lens 增强、MCP 风格协议封装与 Prompt 自动优化机制，能够稳定支持从业务问题到数据结论的端到端自动化分析。

2. 背景与价值

🎯 为什么要做这个项目

企业数据分析普遍存在 SQL 编写门槛高、问题语义理解成本高、跨表分析慢等问题。

🛠️ 解决什么问题

将业务问题自动转化为可执行 SQL，并在失败时自动修复，同时输出结构化结果与可读分析报告。

降低“人工写 SQL、人工排错、人工解释”的时间成本。

👥 适用对象

业务分析师、数据产品经理、数据工程团队、需要快速验证经营问题的运营团队。

3. 项目目标

  • 目标 A：支持自然语言到 SQL 的端到端生成与执行。
  • 目标 B：支持快速路径、慢速路径和仅执行 SQL 模式（fast path / slow path / sql_only），适配不同复杂度问题。
  • 目标 C：支持 SQL 修复、Python 可视化修复、结果分析与流式回传。
  • 目标 D：支持多智能体协作（SQL Agent / Python Agent / Report Agent）与跨 Agent 结果汇总。
  • 目标 E：接入分层 Schema（L0/L1/L2）与 Value Lens，提升复杂问题上下文质量。
  • 目标 F：通过 MCP 风格工具协议统一 Runtime 能力调用与契约管理。
  • 性能目标：单次查询保持可接受端到端时延，SQL 执行阶段优先保持亚秒级到秒级响应。
  • 可靠性目标：构建“生成-执行-修复”闭环，提升复杂场景下的一次成功率与容错能力。
  • 可维护性目标：通过 Kernel / Runtime 分层、契约校验与模块化节点定义降低耦合度。
  • 可观测性目标：SSE 输出 planning / step_start / step_complete / final 事件，支持全链路调试。

4. 设计方案

4.1 整体流程

整体流程

接入提问

用户通过 HTTP/SSE 提交自然语言问题。

路由判断

Kernel Router 判断问题类型与复杂度，决定进入快速路径、慢速路径或仅执行 SQL 模式。

RAG 召回

Runtime Retrieve 基于 RAG（Vector + BM25 + RRF + rerank）召回相关 Schema。

约束生成

Runtime Generate 结合分层 Schema 与 Value Lens 约束生成 SQL。

闭环执行

Runtime Execute 执行 SQL，失败时进入 Runtime Fix 进行修复重试。

分析报告

按用户意图进入可视化、分析报告生成或直接返回 SQL 结果。

流式推送

全过程通过 SSE 实时推送状态和结果，前端展示步骤条、时间线和结果卡片。

4.2 核心组件

组件 职责 说明
接口层（FastAPI） 对外提供聊天接口和流式聊天接口 支持同步返回和 SSE 流式返回
SqlKernel 决策编排：Router / Planner / Reflector / Analyze 不直接执行业务 SQL，专注流程控制
SqlRuntime 检索、生成、执行、修复、可视化 通过 operation 和 adapter 统一对接
RAG 检索器 Schema 召回与排序 Vector + BM25 + RRF + rerank 混合检索
Schema 上下文层 分层 Schema + Value Lens 注入 L0/L1/L2 按任务复杂度动态选择
MCP 风格工具封装 工具协议封装与契约统一 规范输入输出，提升复用性
多智能体协调器 SQL/Python/Report 多角色协作 支持跨 Agent 任务分发与汇总
SSE 事件总线 可观测与调试输出 planning / step_start / step_complete / final

5. 实现说明

5.1 请求上下文说明

用户提交问题时，前端会附带必要的上下文信息，例如会话标识、结果预览行数和是否只执行到 SQL 结果。这样系统可以在同一个请求里完成普通问答、仅执行 SQL 结果和完整分析三种模式。

5.2 SQL 生成说明

系统能够根据自然语言问题自动生成符合业务语义的 SQL。例如处理“某一年内 CZK 支付客户与 EUR 支付客户的消费差值”这类跨表聚合问题时，会自动完成表关联、时间过滤、条件聚合和差值计算，无需人工手写查询语句。

5.3 链路拆分说明

Runtime 层负责把检索、生成、执行、修复、可视化和报告输出拆成不同阶段，节点之间通过结构化状态传递信息。这样做的好处是，每一步都可以独立观察、独立修复，也更容易做后续扩展。

6.3 测试数据与选择动机

本次验证使用的是 BIRD_mini_dev 这类轻量回归数据集。它的作用不是追求全量评测覆盖，而是用较低的调试成本稳定复现 Text-to-SQL 场景中的典型问题。

选择 BIRD_mini_dev 的原因主要有四点：

  • 代表性足够强：它保留了多表连接、时间过滤、聚合统计、条件差值和字段语义对齐等典型 SQL 难点。
  • 迭代成本更低：数据规模更小，适合在开发阶段频繁回归，不会被全量数据拖慢节奏。
  • 便于定位问题：同一批样例可以反复对比检索、生成、执行和修复阶段的变化，方便观察 Prompt 或路由调整带来的影响。
  • 更贴近项目目标：这个项目本来就要验证“自然语言 → SQL → 执行 → 分析”的完整链路，BIRD_mini_dev 很适合做这种链路级验证。

本次截图和结果来自 BIRD_mini_dev 中的示例库 debit_card_specializing。

6.4 测试策略

  • 覆盖范围：链路正确性、SQL 语义正确性、异常修复能力、SSE 事件一致性、前端展示一致性。
  • 验证层次：
    • API 层：输入输出结构、错误码、线程会话字段。
    • Agent 层：路由决策、计划推进、节点执行顺序。
    • Runtime 层：检索、生成、执行、修复、可视化、分析报告。
    • UI 层：步骤条、时间线、结果卡片、状态面板。

6.5 测试环境

  • 操作系统：Windows
  • Python 环境：conda sql-agent
  • 运行方式：项目启动脚本运行主服务
  • 数据来源：BIRD_mini_dev 回归集（示例库：debit_card_specializing）
  • 观察方式：后端日志、SSE 时间线和前端调试控制台

6.6 详细测试过程

详细测试过程

  1. 启动服务并确认健康状态
  • 打开项目后端，确认服务可以正常启动，且健康状态可用。
  • 这一阶段的目标是确保前端、后端、数据库和向量检索基础设施都已准备完毕。
  1. 打开前端调试页并输入测试问题
  • 选择仅执行 SQL 模式，先验证 SQL 生成与执行主链路。
  • 输入问题：What was the difference in gas consumption between CZK-paying customers and EUR-paying customers in 2012?
  1. 观察 SSE 事件序列
  • 期望顺序：planning → step_start(router) → step_complete(router) → step_start(retrieve) → step_complete(retrieve) → step_start(generate) → step_complete(generate) → step_start(execute) → step_complete(execute) → step_start(analyze) → step_complete(analyze) → final
  1. 比对 SQL 结果与参考答案语义
  • 检查生成结果是否正确表达“CZK 消费总量 - EUR 消费总量（2012 年）”。
  • 检查结果是否成功返回且与参考答案语义一致。
  1. 前端结果一致性检查
  • 检查步骤路径、耗时和数据行数是否与时间线一致。
  • 检查 Schema 检索卡片是否展示命中表与 Value Lens。
  • 检查执行结果与分析报告是否与 SQL 语义一致。

补充说明：

  • 该用例属于跨表聚合 + 条件差值问题，能有效验证检索、生成、执行和分析四段链路。
  • 结果与参考 SQL 语义一致，说明关键业务条件（年份、币种、差值方向）没有偏移。
  • 这个 case 很适合展示项目在复杂 SQL 场景下的稳定性，因为它同时包含表关联、时间过滤和条件聚合。

实际测试过程中的关键观察点包括：

  • 问题进入后能否正确识别为数据分析类请求。
  • 检索阶段能否召回 customers、yearmonth 等关键表，并带出有效的 Value Lens 信息。
  • 生成阶段是否能够围绕“差值”这一语义选择正确的聚合方式。
  • 执行阶段是否能稳定返回一行结果，并且耗时保持在可接受范围内。
  • 分析阶段是否能够把结果转化为可读的业务结论，而不是只返回原始数值。

6.8 自动分析报告

下面这段为系统自动生成的分析报告，保留了结论、解读和建议三个层次：

2012 年，使用捷克克朗（CZK）支付的客户与使用欧元（EUR）支付的客户之间的天然气消费量差异为 4.03 亿立方米，具体数值为 402,524,570.17，CZK 支付客户的消费量显著高于 EUR 支付客户。

维度 内容
结果判断 系统正确抓住了“CZK 减 EUR”的差值关系，并且按 2012 年完成过滤。
业务含义 CZK 支付客户是主力消费群体，本地结算市场仍然占据核心位置。
延伸分析 该结果适合继续做年度趋势、客户分层、人均消费和汇率风险分析。

业务建议

优先级 建议 预期效果
高 细分 EUR 客户群体，识别高潜力国际客户。 提升 EUR 收入占比，分散货币风险。
中 评估双币种定价策略，分析消费偏低的原因。 优化定价模型，提升国际竞争力。
中 建立汇率风险对冲思路，降低收入波动。 稳定财务预期，减少汇兑损失。
低 持续跟踪未来年份的差值变化。 观察国际化进程和业务结构变化。
